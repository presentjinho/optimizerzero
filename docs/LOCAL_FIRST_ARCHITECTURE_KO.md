# OptimizerZero local-first 아키텍처

OptimizerZero의 최적 방향은 서버 저장 없는 정적 웹앱이다.

## 결론

- 메인 배포: Cloudflare Pages
- 백업 배포: Netlify
- 앱 형태: 정적 Web Lite + PWA
- 파일 처리: 브라우저 내부
- 사용자 파일 저장: 서버 저장 없음
- 원본 파일 보관: 의도적으로 하지 않음
- 무거운 작업: 데스크톱 버전으로 안내

이 구조가 OptimizerZero에 가장 맞는 이유는 압축 대상 파일이 개인 문서, 만화, 이미지, Office 파일일 수 있기 때문이다. 서버 업로드를 만들면 기능은 편해져도 신뢰도가 떨어진다.

## 지금 선택하지 않는 것

- 중앙 DB: 사용하지 않는다.
- Supabase/Firebase 같은 사용자 데이터 DB: 사용하지 않는다.
- 서버 업로드 압축: 사용하지 않는다.
- 서버 로그 기반 분석: 사용하지 않는다.
- 로그인 필수 구조: 사용하지 않는다.
- Google Drive 동기화: 지금은 넣지 않는다.

## 나중에만 고려할 것

여러 기기 동기화가 꼭 필요해질 때만 다음을 검토한다.

- IndexedDB 또는 OPFS에 설정/작업 기록만 저장
- 사용자가 직접 고른 개인 클라우드에만 저장
- Web Crypto API로 먼저 암호화
- 서버에는 평문 파일이나 평문 기록을 두지 않음

파일 압축 자체는 계속 브라우저 또는 데스크톱에서 처리한다.

## 배포 구조

```text
사용자 브라우저
  ├─ 파일 선택
  ├─ 압축/검증/리포트 생성
  └─ 결과 다운로드

Cloudflare Pages / Netlify
  └─ 정적 앱 파일만 제공
```

서버는 앱 코드만 제공한다. 선택한 파일과 결과물은 OptimizerZero 서버로 올라가지 않는다.

## 기능 추가 원칙

- 파일 업로드 기능을 추가하지 않는다.
- 분석/추적 스크립트를 추가하지 않는다.
- 외부 CDN 의존성을 추가하지 않는다.
- 새 기능이 네트워크를 쓰면 개인정보 문서를 먼저 수정한다.
- 사용자가 직접 선택한 저장 위치 외에는 원본 파일을 저장하지 않는다.
- 브라우저 메모리 한계를 넘는 작업은 데스크톱 버전으로 안내한다.

## 최적 선택

현재 OptimizerZero에는 다음 조합이 가장 좋다.

- Vanilla JS Web Lite
- Cloudflare Pages
- `web/vendor/` 로컬 의존성
- Service Worker PWA 캐시
- Cloudflare Access로 비공개 테스트
- 공개 전환 시에도 no analytics/no upload 유지
